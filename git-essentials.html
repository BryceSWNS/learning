环境配置 & 基础概念

入职第一件事：配置好身份信息，避免提交记录混乱。

# 配置用户信息（全局）
git config --global user.name  "张三"
git config --global user.email "zhangsan@company.com"

# 配置默认编辑器（推荐 VS Code）
git config --global core.editor "code --wait"

# 配置默认分支名为 main
git config --global init.defaultBranch main

# 查看当前所有配置
git config --list

三个工作区域

Working Directory工作区
→
git add暂存区 Index
→
git commit本地仓库
→
git push远程仓库
提示：公司项目通常使用 SSH Key 认证。入职后先配置 ~/.ssh/id_ed25519，并将公钥添加到 GitLab/GitHub 账户。

日常开发流程

大厂每天最高频的操作，必须形成肌肉记忆。

# 查看状态（改动了哪些文件）
git status

# 查看具体改动内容
git diff                   # 工作区 vs 暂存区
git diff --staged          # 暂存区 vs 上次提交

# 暂存文件
git add src/feature.ts     # 指定文件
git add -p                # 交互式选择代码块（推荐！）
git add .                  # 所有改动（谨慎用）

# 提交
git commit -m "feat: 新增用户登录功能"

# 查看提交历史
git log --oneline --graph --decorate  # 推荐加这些参数
📋

git status

随时查看工作区状态，是最常用的命令之一，没有副作用，放心用。

🔍

git diff -p

交互式暂存，可以只提交同一个文件里的部分改动，保持提交原子性。

📜

git log --oneline

简洁历史视图，加 --graph 可以看分支合并图，非常直观。

分支管理策略

大厂通常采用 Git Flow 或 Trunk-Based Development 策略，理解分支模型是协作的前提。

常用分支命令

# 查看分支
git branch                            # 本地分支
git branch -a                         # 包括远程分支

# 创建 & 切换（推荐写法）
git switch -c feature/user-login      # 新建并切换
git switch main                       # 切换已有分支

# 老写法（仍然常见）
git checkout -b feature/user-login

# 删除分支
git branch -d feature/user-login      # 已合并才能删
git branch -D feature/user-login      # 强制删除

大厂分支命名规范

前缀用途示例
feature/新功能开发feature/user-login
fix/Bug 修复fix/login-crash-ios
hotfix/生产紧急修复hotfix/payment-null-pointer
release/版本发布准备release/v2.3.0
chore/工具/依赖更新chore/upgrade-react-18
refactor/重构（不改功能）refactor/extract-auth-module

远程仓库协作

# 拉取最新代码（推荐）
git pull --rebase origin main   # rebase 方式，历史更整洁

# 等价于以下两步
git fetch origin
git rebase origin/main

# 推送分支
git push origin feature/user-login
git push -u origin feature/user-login  # 设置上游，之后只需 git push

# 查看远程信息
git remote -v

# 从远程拉取新分支到本地
git fetch origin
git switch feature/other-dev-branch  # 自动跟踪远程
最佳实践：开始一天工作前，先 git pull --rebase 同步主干，避免本地积累太多分歧，PR 时冲突爆炸。

提交规范（Conventional Commits）

大厂的 CI/CD 流水线、CHANGELOG 自动生成都依赖规范的 commit message。

# 格式：<type>(<scope>): <subject>

feat(auth): 新增 OAuth2 第三方登录支持
fix(cart): 修复商品数量为 0 时仍可结算的问题
docs(api): 更新用户接口文档
style(home): 统一首页按钮圆角为 8px
refactor(db): 将 MySQL 查询抽象为 Repository 层
test(order): 补充订单状态机单元测试
chore(deps): 升级 Spring Boot 至 3.2.0
perf(search): 添加商品搜索结果缓存，P99 降低 60%

Type 速查

Type含义
feat新功能（会出现在 CHANGELOG）
fixBug 修复（会出现在 CHANGELOG）
docs仅文档变更
style代码格式，不影响逻辑
refactor重构，既非新功能也非修 bug
test添加或修改测试
chore构建过程、辅助工具变更
perf性能优化
ciCI 配置变更
注意：破坏性变更（Breaking Change）需在 type 后加 !，如 feat!: 重命名用户 API 路径，或在 body 中写 BREAKING CHANGE: ...。

合并 & 变基

Merge vs Rebase

🔀

git merge

保留完整历史，生成 merge commit。多人协作的公共分支用这个，历史可追溯。

📐

git rebase

线性历史，更整洁。个人 feature 分支同步主干时推荐，不要对已推送的公共分支 rebase。

# 场景：feature 分支同步 main 最新代码
git switch feature/user-login
git rebase main              # 将 feature 的提交接到 main 顶端

# 解决冲突后继续
# 1. 编辑冲突文件
git add <冲突文件>
git rebase --continue

# 中途想放弃 rebase
git rebase --abort

# 交互式 rebase：整理提交历史（PR 前清理）
git rebase -i HEAD~3          # 整理最近 3 个提交
# pick  = 保留
# squash = 合并到上一个
# reword = 修改提交信息
黄金法则：永远不要对已经推送到远程的公共分支（main、develop）执行 rebase，会导致其他人历史混乱。

代码审查 & Pull Request

大厂所有代码必须经过 PR + Code Review 才能合入主干，这是质量保障的核心环节。

提 PR 前自检

# 1. 同步主干，解决冲突
git fetch origin
git rebase origin/main

# 2. 清理提交历史（squash 零散 WIP 提交）
git rebase -i origin/main

# 3. 查看本次 PR 包含哪些改动
git diff origin/main...HEAD
git log origin/main...HEAD --oneline

# 4. 推送并创建 PR
git push -u origin feature/user-login

Review 中 reviewer 要求修改

# 在原分支上继续提交修改
git add .
git commit -m "fix: 根据 review 意见修复 token 过期处理"
git push                          # PR 自动更新

# 如果需要修改最后一次提交内容（未推送时）
git commit --amend --no-edit     # 代码修改并入上一条提交
git push --force-with-lease       # 比 --force 更安全
提示：使用 --force-with-lease 而非 --force，前者会检查远程是否有你不知道的新提交，避免覆盖别人的工作。

撤销 & 回滚

操作失误时冷静选择正确命令，避免越搞越乱。

场景命令说明
撤销工作区修改（未 add） git restore <file> 文件回到上次提交状态
撤销暂存（已 add 未 commit） git restore --staged <file> 从暂存区移出，改动保留
修改最后一次提交 git commit --amend 仅用于未推送的提交
撤销某次提交（安全）推荐 git revert <commit-hash> 生成新的"撤销提交"，历史保留
回退到某次提交（危险）危险 git reset --hard <hash> 丢弃之后所有提交，不可逆
紧急回滚已发布版本 git revert HEAD 最安全的生产回滚方式 
# 安全回滚：生产故障时用 revert
git revert abc1234               # 撤销特定提交
git revert HEAD~2..HEAD          # 撤销最近 2 个提交
git push                          # 推送回滚提交

# reset 三种模式（慎用）
git reset --soft  HEAD~1         # 撤销提交，改动留在暂存区
git reset --mixed HEAD~1         # 撤销提交，改动留在工作区（默认）
git reset --hard  HEAD~1         # 撤销提交，改动全部丢弃 ⚠️
生产事故原则：已推送到主干的 commit 只用 git revert，绝对不用 reset --hard + force push。

进阶技巧

stash — 临时搁置

# 场景：正在开发，突然要切换分支处理 bug
git stash push -m "WIP: 用户登录页半成品"
git switch fix/urgent-bug

# 处理完 bug 回来恢复
git switch feature/user-login
git stash list                   # 查看所有 stash
git stash pop                    # 恢复最新 stash 并删除
git stash apply stash@{1}        # 恢复指定 stash（保留）

cherry-pick — 挑选提交

# 场景：把 feature 分支上某个 commit 应用到 hotfix 分支
git switch hotfix/urgent
git cherry-pick abc1234          # 挑选特定提交
git cherry-pick abc1234..def5678 # 挑选一段范围

bisect — 二分查找问题提交

# 场景：不知道哪次提交引入了 bug
git bisect start
git bisect bad                   # 当前版本有 bug
git bisect good v1.2.0           # 这个版本没问题
# Git 自动二分，你测试后输入 good/bad
git bisect good                  # 或 git bisect bad
# 找到问题提交后
git bisect reset                 # 恢复原始状态

常用 alias 配置

git config --global alias.lg  "log --oneline --graph --decorate --all"
git config --global alias.st  "status -sb"
git config --global alias.co  "switch"
git config --global alias.undo "reset --soft HEAD~1"

# 之后可以这样用
git lg     # 漂亮的分支图
git undo   # 安全撤销最后一次提交

常见事故处理

误删分支 / 误 reset

# reflog 记录了所有 HEAD 的移动历史，是 Git 的"后悔药"
git reflog
# 输出类似：
# abc1234 HEAD@{0}: reset: moving to HEAD~3
# def5678 HEAD@{1}: commit: feat: 新增支付功能

# 恢复到误操作前的状态
git reset --hard def5678

# 恢复误删的分支
git branch feature/recovered def5678

提交了敏感信息（密码/密钥）

# 如果还没推送：修改最后一次提交
git commit --amend

# 如果已推送：立即做这两件事
# 1. 吊销/轮换密钥（最重要！！！）
# 2. 使用 git filter-repo 从历史中彻底删除
pip install git-filter-repo
git filter-repo --path secrets.env --invert-paths
记住：一旦推送到远程，就应假设密钥已泄露，必须先吊销密钥，清理历史是第二步。

合并冲突不知道怎么解

# 冲突文件长这样：
# <<<<<<< HEAD（你的改动）
# const timeout = 5000;
# =======
# const timeout = 3000;
# >>>>>>> feature/optimize（对方的改动）

# 使用 mergetool 可视化解决
git mergetool                    # 打开配置的可视化工具

# 或者直接编辑文件后
git add <解决后的文件>
git rebase --continue           # 或 git merge --continue

# 实在不确定，放弃本次操作
git rebase --abort
git merge  --abort
